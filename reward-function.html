Introduction

In reinforcement learning, an agent (the robot) observes a state s, takes an action a, transitions to a new state s′, and receives a reward scalar from the environment. The agent's single objective is to maximize the expected sum of discounted rewards over time — the return.

The reward function R(s, a, s′) is the most influential design decision in a robotics RL system. It encodes what success means. Get it right and an intelligent behavior emerges. Get it wrong and the agent exploits loopholes you never imagined.

Key Insight

The agent will find exactly what you measure — not what you intend. This is the fundamental challenge of reward design. The robot is a perfect optimizer; it will find every gap between your reward proxy and your true goal.

The RL Loop

Every robotics RL system follows this cycle: the robot observes the world, chooses an action through its policy π(a|s), receives a reward signal, and updates its policy via a learning algorithm (PPO, SAC, TD3, etc.).

Python Basic RL loop 
# The fundamental reward signal flow
def reward_function(state, action, next_state):
    # Define what "good" behavior means
    r_task    = 1.0 * forward_velocity(next_state)
    r_energy  = -0.1 * np.sum(action ** 2)
    r_survive = 1.0 * is_alive(next_state)
    r_safety  = -0.05 * contact_forces(next_state)
    return r_task + r_energy + r_survive + r_safety

# Training loop
for episode in range(num_episodes):
    state = env.reset()
    while not done:
        action = policy.act(state)           # π(a|s)
        next_state, _, done, _ = env.step(action)
        reward = reward_function(state, action, next_state)
        policy.update(state, action, reward, next_state)
        state = next_state

Core Principles

Principle 1 — Specify Goals, Not Paths

Reward the outcome, not the mechanism. If you reward "lift arm to 90°", the robot learns that exact pose — not "pick up objects." This over-specification prevents the agent from discovering better strategies.

❌ Behavior-Specified (Brittle)
Reward: lift_angle > 90°

Robot holds arm at 91° indefinitely. Never learns grasping. Brittle to object position changes.

✓ Goal-Specified (Robust)
Reward: object_height > threshold

Robot discovers multiple grasping strategies. Generalizes to new object positions. Emergent behavior.

Principle 2 — Reward Shaping

A well-shaped reward guides learning without over-constraining it. The shaping function F(s, s′) adds potential-based guidance while preserving the optimal policy:

Math Potential-based shaping (Ng et al., 1999) 
# Shaped reward preserves optimal policy
R_shaped(s, a, s′) = R(s, a, s′) + γ·Φ(s′) - Φ(s)

# Example: use distance-to-goal as potential
Φ(s) = -distance_to_goal(s)

# Result: agent gets small rewards for getting closer
# without changing which final policy is optimal

Principle 3 — Multi-Objective Balancing

Real robots must balance competing objectives simultaneously. The weighted sum approach:

R =  w₁·R_task  + w₂·R_smooth  − w₃·R_energy  − w₄·R_danger

Tuning the weights wᵢ is the primary engineering challenge — small changes produce drastically different emergent behaviors.

03

Sparse vs Dense Rewards

One of the first design decisions: how often does the robot receive feedback? This is the sparsity tradeoff.

Sparse Reward

  • Signal only at goal completion
  • Clean, unambiguous objective
  • Hard to explore from — credit assignment is difficult
  • Requires HER, curriculum learning, or demonstrations
  • Better generalization when it works

Dense Reward

  • Signal at every timestep
  • Fast learning signal, easy to explore
  • High risk of reward hacking
  • May over-shape toward proxy behavior
  • Default choice for most locomotion tasks
The Exploration Problem with Sparse Rewards

If a robot must perform 20 correct steps to reach a goal and receive any reward, random exploration almost never reaches it. Techniques like Hindsight Experience Replay (HER), curiosity-driven exploration, and reward curriculum help bridge this gap.

04

Reward Hacking

Reward hacking occurs when the agent discovers unintended ways to maximize the reward proxy that violate the spirit of the task. It is inevitable if your reward function has any gap between the proxy and the true goal.

Task Intended Behavior Exploited Behavior Cause
Locomotion Run forward fast Stand still (high alive bonus) Weight imbalance
Grasping Pick up object Push object off table (episode ends = no penalty) Missing penalty
Sim locomotion Walk on terrain Exploit physics glitches to fly Sim artifacts
Manipulation Sort colored blocks Cover camera (appears sorted) Proxy measure
Goodhart's Law in RL

"When a measure becomes a target, it ceases to be a good measure." — Once the agent optimizes your reward proxy hard enough, it diverges from your true intent. Always ask: what is the worst possible way to maximize this reward?

05

Multi-Objective Reward Design

Production robotics tasks require balancing multiple objectives simultaneously. Common components and their engineering rationale:

Component Type Typical Range Engineering Purpose
R_task Positive w ∈ [1, 5] Primary goal signal. Progress toward objective.
R_control Penalty w ∈ [0.001, 0.5] Penalize joint torques. Prevents motor wear on real hardware.
R_alive Positive w ∈ [0.5, 2] Survival bonus. Keep robot upright and operational.
R_contact Penalty w ∈ [0.01, 0.5] Penalize ground impact forces. Prevents hardware damage.
R_smooth Positive w ∈ [0.01, 0.3] Reward smooth trajectories. Critical for sim-to-real transfer.
06

Case Study: OpenAI Half-Cheetah

The MuJoCo Half-Cheetah is the most studied reward function in locomotion RL. Introduced by Wierstra et al. and popularized through OpenAI Gym, it has been the benchmark for hundreds of RL papers including PPO, SAC, TRPO, TD3, and more.

The robot is a 2D planar cheetah with 6 actuated joints. The task: run forward as fast as possible while being energy-efficient and staying alive.

Why It's Famous

The Half-Cheetah reward is elegant because it produces a naturally-looking galloping gait using only three simple terms. The agent was never told how to run — it discovered that behavior from pure optimization pressure.

07

Formula Breakdown

The canonical Half-Cheetah reward from MuJoCo (used in most RL benchmarks):

R =  +v_x  − 0.1 · ‖a‖²  + 1.0
+ v_x
Forward velocity reward. Proportional to x-axis speed.
− 0.1·‖a‖²
Control cost. L2 norm of joint torques penalized.
+ 1.0
Alive bonus. Constant per timestep for staying upright.
Python HalfCheetah-v4 reward — from OpenAI Gymnasium source 
def step(self, action):
    x_position_before = self.data.qpos[0]
    self.do_simulation(action, self.frame_skip)
    x_position_after  = self.data.qpos[0]

    # dt = timestep × frames per step
    x_velocity = (x_position_after - x_position_before) / self.dt

    # ── Reward components ──────────────────────────
    forward_reward = self._forward_reward_weight * x_velocity
    ctrl_cost      = self.control_cost(action)       # 0.1 * sum(a²)

    # Total reward
    reward = forward_reward - ctrl_cost

    # Note: alive_bonus removed in v4 (implicit in no termination)
    return observation, reward, terminated, truncated, info

def control_cost(self, action):
    return self._ctrl_cost_weight * np.sum(np.square(action))

Effect of Weight Changes

Configuration Alive Bonus Velocity Weight Control Cost Emergent Behavior
Default 1.0 1.0 0.1 Natural gallop, energy-efficient
Broken 5.0 0.0 0.0 Stands perfectly still (reward hacked)
Speed demon 0.5 5.0 0.0 Chaotic thrashing, fast but violent
Efficient 1.0 1.0 1.0 Barely moves — energy cost dominates
Cautious 3.0 0.5 0.5 Slow careful steps, avoids risk

Reward Design Process

A structured workflow for designing reliable reward functions in real robotics projects:

1

Define success unambiguously

Write a precise, measurable definition of task success. Avoid vague goals. "Robot picks up object and places it in bin" — not "robot does the task well."

2

Choose signal density

Decide sparse vs dense. For manipulation with clear terminal states: try sparse first. For locomotion and continuous control: start dense. Use shaping if sparse is too hard to explore.

3

Add necessary penalty terms

Ask: what are the top 3 ways to maximize this reward without actually doing the task? Add penalty terms to block each exploit. Repeat after each training run.

4

Set initial weights via domain knowledge

Use physics intuition. If forward velocity is ~2 m/s and control cost terms are ~0.3, a weight of 0.1 keeps them comparable. Normalize or scale terms before tuning.

5

Train and audit for exploits

Watch rollout videos, not just reward curves. Agents are creative. Log each component separately to understand which term is driving behavior.

6

Iterate and document

Track every reward function change with its motivation and result. Reward engineering is empirical science. Version control your reward function like code.

10

Common Pitfalls

Pitfall Symptom Fix
Unscaled terms One reward component dominates entirely Normalize all terms to similar magnitude before adding weights
Missing terminal condition Agent learns to end episodes early to avoid penalties Add episode-end penalty or remove terminal conditions that can be exploited
Conflicting objectives Agent oscillates between behaviors, unstable training Diagnose via per-component reward logging; adjust weights or decouple objectives
Alive bonus too high Agent learns to stand still Reduce alive bonus or replace with task-progress bonus
Proxy measure Agent achieves metric without task (Goodhart's Law) Use learned reward models, RLHF, or multi-angle success criteria
Sim-only rewards Great sim performance, poor real transfer Add smoothness penalties; penalize high-frequency control; reduce sim speed bonuses
Advanced: Learned Reward Models

When hand-engineering reward functions becomes intractable, consider Inverse Reinforcement Learning (IRL) to infer rewards from demonstrations, or RLHF (Reinforcement Learning from Human Feedback) to learn a reward model from human preference comparisons — the same technique used in modern LLMs like GPT-4.
