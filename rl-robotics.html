01

Why Reinforcement Learning in Robotics?

Traditional robot control relies on classical control theory — PID controllers, model predictive control (MPC), trajectory planning. These work well when you can fully model the system. But robots operate in the real, messy, unpredictable world.

⚙️ Classical Control

  • Requires accurate mathematical model of the system
  • Works well for linear, well-defined dynamics
  • Brittle when facing unexpected perturbations
  • Hard to generalize across tasks
  • Expert knowledge required for every new scenario

🧠 Reinforcement Learning

  • Learns control from experience, no perfect model needed
  • Handles nonlinear, high-dimensional dynamics
  • Can adapt and generalize
  • Discovers novel strategies humans wouldn't design
  • End-to-end learning (sensor → action)
🤖 Robotics Context

Think of a quadruped robot learning to walk. Writing a controller for 12 joints, accounting for ground contacts, CoM shifts, and disturbances is an enormous modeling challenge. RL can learn this entirely from trial and error in a simulator — and transfer it to the real world.

🎯

Core Idea of RL

An agent (robot) interacts with an environment (physical world) by taking actions, receiving rewards, and updating its policy to maximize long-term cumulative reward. No explicit teacher — just feedback from the world.

02

Markov Decision Process (MDP)

Everything in RL is built on the Markov Decision Process — the mathematical framework that formalizes how an agent interacts with an environment over time.

📐 Formal Definition

An MDP is defined by the tuple: M = (S, A, P, R, γ)

S — State Space

All possible situations the agent can be in.

// Quadcopter state example
s = [x, y, z, // position
     φ, θ, ψ, // roll, pitch, yaw
     vx, vy, vz, // linear velocity
     ωx, ωy, ωz] // angular velocity

A 12-dimensional continuous state space for a drone.

A — Action Space

All actions the agent can take.

// Quadcopter action
a = [ω₁, ω₂, ω₃, ω₄]
// motor speeds (RPM)

// Or higher-level:
a = [thrust, roll_cmd,
     pitch_cmd, yaw_cmd]

P — Transition Dynamics

P(s' | s, a)

// Probability of landing
// in state s' given we
// were in s and took a

In real robots, this is rarely known exactly — this is why we use model-free RL.

R — Reward Function

R(s, a, s')

// Drone hovering reward:
r = -||pos - target||²
   - 0.1 * ||ω||² // penalize motor speed
   - 10 * crash // big penalty if crash

γ — Discount Factor

G_t = r_t + γ·r_{t+1} + γ²·r_{t+2} + ... + γⁿ·r_{t+n}

// γ ∈ [0, 1]
// γ → 0: myopic agent (only cares about immediate reward)
// γ → 1: far-sighted agent (cares about distant future)
// Typical: γ = 0.99 in robotics tasks

The discount factor prevents infinite sums and encodes how much the agent cares about future vs immediate rewards. For a drone stabilization task, γ = 0.99 means a reward 100 steps away is still worth ~37% of a reward now.

🔑 The Markov Property

The "Markov" in MDP means: the future depends only on the current state, not the past. Formally: P(s_{t+1} | s_t, a_t) = P(s_{t+1} | s_0,...,s_t, a_0,...,a_t). This is why the state must contain ALL information needed to make a decision — for a drone, that means including velocity, not just position.

03

Value Functions & Bellman Equations

The central question in RL is: "How good is it to be in state s?" or "How good is action a in state s?". Value functions answer this.

V(s) — State Value Function

Expected cumulative reward starting from state s and following policy π.

V^π(s) = E_π[G_t | s_t = s]

= E_π[Σ γᵏ r_{t+k} | s_t = s]

// "How good is this state?"

Example: A drone at stable hover altitude has high V(s). A drone in freefall has very negative V(s).

Q(s,a) — Action-Value Function

Expected cumulative reward starting from state s, taking action a, then following π.

Q^π(s,a) = E_π[G_t | s_t=s, a_t=a]

= R(s,a) + γ Σ P(s'|s,a) V^π(s')

// "How good is this action?"

The Q-function is the most directly useful: it tells the agent which action to take.

🔄 Bellman Expectation Equation

The most fundamental equation in RL — V(s) can be decomposed recursively:

V^π(s) = Σ_a π(a|s) [R(s,a) + γ Σ_{s'} P(s'|s,a) V^π(s')]

────────────────────────────────────────────────
Current reward + γ × future value

Q^π(s,a) = R(s,a) + γ Σ_{s'} P(s'|s,a) Σ_{a'} π(a'|s') Q^π(s',a')

Think of it like: "the value of where I am now = what I get immediately + the discounted value of where I'll end up."

⭐ Bellman Optimality Equation

For the optimal policy π*, the Bellman optimality equation gives the best possible value:

V*(s) = max_a [R(s,a) + γ Σ_{s'} P(s'|s,a) V*(s')]

Q*(s,a) = R(s,a) + γ Σ_{s'} P(s'|s,a) max_{a'} Q*(s',a')

// Optimal policy: π*(s) = argmax_a Q*(s,a)
🤖 Robotics Insight

If we knew Q*(s,a) perfectly, controlling a robot is trivial: at every state s, just pick the action with the highest Q value. All of deep RL is fundamentally about approximating Q* or π* when the state and action spaces are too large for exact computation.

04

Policy: The Robot's Brain

The policy π is the mapping from states to actions — it IS the controller. Everything in RL aims to find a good policy.

📊 Deterministic Policy

π: S → A
a = π(s)

// "In state s, always do action a"
// Used in: DDPG, TD3

Better for continuous robot actions like joint torques. Faster at inference.

🎲 Stochastic Policy

π: S → P(A)
a ~ π(a|s)

// "Sample action from distribution"
// Used in: PPO, SAC, TRPO

Better for exploration and handling uncertainty. Natural for contact-rich manipulation.

Three Approaches to Finding π*

Value-Based

Learn Q*(s,a), then act greedily. Best for discrete actions.

Policy-Based

Directly optimize π. Best for continuous action spaces (robotics).

05

Classical RL Algorithms

Before deep learning entered the picture, classical RL used tabular methods or linear function approximation. Understanding these is essential before jumping to deep RL.

📋 Q-Learning (Tabular)

The most foundational off-policy, value-based algorithm. Stores Q(s,a) as a table.

// Q-learning update rule (Temporal Difference)
Q(s,a) ← Q(s,a) + α [r + γ max_{a'} Q(s',a') - Q(s,a)]

TD error (δ) = r + γ max Q(s',a') - Q(s,a)

// α = learning rate (e.g. 0.01)
// The target: r + γ max Q(s', a') is the Bellman target
⚠️ Why tabular Q-learning breaks in robotics

A 12D drone state with continuous values → infinite number of states. You'd need a table with infinite rows. This is the fundamental problem deep RL solves: use a neural network to generalize across states.

🎲 Exploration: ε-Greedy

The classic exploration-exploitation dilemma: how does the robot try new things vs exploit what it knows?

With probability ε: take random action (explore)
With probability 1-ε: take a = argmax Q(s,a) (exploit)

// Decaying schedule: ε = 1.0 → 0.05 over training
// Start with lots of exploration, then exploit more

🔁 SARSA vs Q-Learning

Property Q-Learning SARSA
Type Off-policy On-policy
Update target max Q(s', a') — best possible Q(s', a') — action actually taken
Safer? No Yes
Robotics use Simulation training Real hardware (safer)

In robotics: SARSA is safer for real hardware because it accounts for the actual exploratory behavior. Q-learning can be reckless.

06

The Deep RL Revolution

Deep RL replaces the Q-table with a neural network — a universal function approximator. Now we can handle the continuous, high-dimensional states that robotics requires.

🧠 DQN — The Bridge (2013, DeepMind)

Deep Q-Network was the first major success. Proved that NNs could learn Q-functions stably.

Q(s, a; θ) ← Neural Network with params θ

// Loss: minimize Bellman error
L(θ) = E[(r + γ max_{a'} Q(s',a'; θ⁻) - Q(s,a; θ))²]

// Two key tricks:
1. Experience Replay Buffer D = {(s,a,r,s')}
2. Target Network θ⁻ (updated slowly)
🗄️

Why Experience Replay?

Neural networks assume i.i.d. (independent) data. But consecutive (s,a,r,s') tuples are highly correlated. Replay buffers store millions of past transitions and sample mini-batches randomly — breaking the correlation and dramatically stabilizing training.

⚠️ DQN Limitation for Robotics

DQN only works with discrete action spaces. Robot joints require continuous torques — you can't discretize "apply exactly 3.47 Nm to joint 3." This drove the development of continuous-action deep RL algorithms.

🎭 Policy Gradient — The Foundation of Modern RL

Instead of learning Q and acting greedily, directly parameterize and optimize the policy π_θ.

// Objective: maximize expected return
J(θ) = E_{τ ~ π_θ}[G(τ)] // τ = trajectory

// Policy Gradient Theorem:
∇_θ J(θ) = E[∇_θ log π_θ(a|s) · G_t]

∇_θ log π_θ(a|s) ← "score function"
G_t ← return (reward signal)

Intuition: if an action led to high return, increase its probability. If it led to low return, decrease it.

REINFORCE (Monte Carlo PG)

Collect full episode, compute returns, update θ. Simple but very high variance. Rarely used alone in robotics.

Actor-Critic (A2C/A3C)

Replace G_t with Q(s,a) or advantage A(s,a) = Q(s,a) - V(s). Dramatically reduces variance. Foundation for PPO and SAC.

📊 Advantage Function — Critical for Robotics

A(s, a) = Q(s, a) - V(s)

// "How much BETTER is action a compared to
// the average action in state s?"

// Practical estimate (GAE - Generalized Advantage Estimation):
Â_t = Σ_{l=0}^{∞} (γλ)ˡ δ_{t+l}
// where δ_t = r_t + γV(s_{t+1}) - V(s_t)
// λ ∈ [0,1] controls bias-variance tradeoff

GAE (Schulman 2016) is used in almost every modern robotics RL paper. λ=0.95 is a common default.

07

PPO — Proximal Policy Optimization

🏆 Why PPO dominates robotics RL

PPO is the workhorse of robotics locomotion. Used in OpenAI's locomotion work, Agility Robotics (Cassie), Boston Dynamics research, NVIDIA Isaac Gym. It's stable, sample-efficient enough, and works well out of the box.

PPO solves a key problem with vanilla policy gradients: if you take too large an update step, the policy can collapse catastrophically. In robotics, this means the robot falls over and never recovers.

🔒 The Core Problem PPO Solves

TRPO (Trust Region Policy Optimization) solved this first by constraining the KL divergence between old and new policy — but was computationally expensive. PPO achieves the same stability with a simple clipping trick.

// Define probability ratio:
r_t(θ) = π_θ(a_t|s_t) / π_{θ_old}(a_t|s_t)

// Vanilla PG objective (can cause huge steps):
L^PG = E[r_t(θ) · Â_t]

// PPO Clipped Objective:
L^CLIP = E[min(r_t(θ)·Â_t, clip(r_t(θ), 1-ε, 1+ε)·Â_t)]

// ε = 0.2 (common default)
// This CLIPS the update if r_t strays too far from 1

Understanding the PPO Clip Intuitively

When Â > 0 (good action)

We want to increase π(a|s). But we clip at (1+ε) — don't increase probability too aggressively. The bound prevents overshooting.

When Â < 0 (bad action)

We want to decrease π(a|s). But we clip at (1-ε) — don't decrease probability too aggressively. Prevents the policy from unlearning good behaviors.

📝 PPO Full Algorithm — Robotics Implementation

// Full PPO loop for a robot:

Initialize policy π_θ, value net V_φ

for iteration = 1, 2, ... do:
  // 1. Collect N timesteps of robot experience
  D = {s_t, a_t, r_t, s_{t+1}} from π_θ_old

  // 2. Compute advantages using GAE
  Â_t = GAE(r_t, V_φ(s_t)) with λ=0.95

  // 3. Optimize for K epochs (e.g. K=10)
  for epoch in range(K):
    θ ← θ + α ∇_θ L^CLIP(θ)
    φ ← φ - α ∇_φ (V_φ(s) - V_target)²

  θ_old ← θ
end for
Network

MLP: 2-3 hidden layers, 256-512 units, tanh activation for robot control

Rollout size

Typically 2048-4096 steps per update. More steps = lower variance but slower updates.

Parallelism

Run 16-4096 parallel environments in sim. Isaac Gym runs 4096 robots simultaneously on GPU.

🤖 PPO in Robot Locomotion — Reward Shaping Example

// Legged robot locomotion reward (typical):
r = w₁ · v_forward // reward forward velocity
  - w₂ · ||τ||² // penalize joint torques
  - w₃ · ||ω_body||² // penalize body angular velocity
  - w₄ · foot_contact_force // smooth foot contacts
  - w₅ · action_rate // penalize jittery actions
  + w₆ · alive_bonus // stay upright

Reward shaping is an art — each term guides behavior. This is one of the most critical (and difficult) parts of applying RL to real robots.

SAC — Soft Actor-Critic

🔥 SAC's unique contribution

SAC introduces maximum entropy RL — the agent is encouraged to be as random as possible while still performing well. This makes it extraordinarily robust for robot manipulation and contact-rich tasks.

🌡️ Maximum Entropy Framework

SAC doesn't just maximize reward — it maximizes reward PLUS policy entropy H(π):

J(π) = Σ_t E[r(s_t, a_t) + α · H(π(·|s_t))]

// H(π) = -Σ π(a|s) log π(a|s) ← entropy
// α = temperature parameter (can be auto-tuned)

// High entropy = more random = more exploration
// The agent explores naturally, without ε-greedy

For robotics, this means: SAC explores better in manipulation tasks with many local optima, and the resulting policy is more robust to perturbations.

🏗️ SAC Architecture

// SAC has 5 neural networks:
π_θ(a|s) ← Actor (policy) — Gaussian output
Q_φ1(s,a) ← Critic 1 (Q-function)
Q_φ2(s,a) ← Critic 2 (Q-function)
Q_φ1_target(s,a) ← Target Critic 1 (slow update)
Q_φ2_target(s,a) ← Target Critic 2 (slow update)

// Two critics → take min → prevents Q overestimation
// (Double Q-learning trick)

SAC Soft Bellman Target

y = r + γ · (min(Q₁_target, Q₂_target)(s',ã') - α log π(ã'|s'))
// ã' ~ π(·|s') — sample next action from policy

🎛️ Automatic Temperature Tuning

One of SAC's killer features — α adjusts automatically so the entropy stays at a target value:

min_α E[-α (log π(a|s) + H_target)]

// H_target = -|A| (target entropy = negative action dim)
// For a 6-DOF robot arm: H_target = -6
// α increases if entropy too low → more exploration
// α decreases if entropy too high → more exploitation

Other Important Algorithms

TD3 (Twin Delayed DDPG) — deterministic policy, great for low-noise environments. DDPG — first continuous action deep RL, ancestor of SAC and TD3. Dreamer / DreamerV3 — model-based RL, learns a world model first, extremely sample efficient for robotics.

Sim-to-Real Transfer

Training on real hardware is expensive, dangerous, and slow. We train in simulation, then deploy on real robots. But simulators aren't perfect — this creates the reality gap.

🌍 The Reality Gap Problem

  • Simulator physics ≠ real physics (friction, motor delays, flex, etc.)
  • Sensor noise in real world that doesn't exist in sim
  • Unmodeled dynamics (cable sag, IMU bias, thermal effects)
  • Policy trained in perfect sim fails when deployed

🎲 Domain Randomization

Randomize simulation parameters during training so the policy learns to be robust to variation.

// Randomize at each episode:
mass ∈ [0.8×, 1.2×] nominal
friction ∈ [0.5, 1.5]
motor_delay ∈ [0, 20ms]
PD_gains ∈ [0.8×, 1.2×]
external_force ~ N(0, 5N)

If the policy works across all these variations, real-world deployment is just another domain.

🔧 System Identification

Measure real robot parameters precisely and narrow the sim gap before training.

  • Motor torque curves from hardware tests
  • Inertia tensors from CAD
  • Joint friction estimation
  • Contact dynamics calibration

Quadcopter Control: From PID to RL — The RAPTOR Story

⚡ Why Quadcopters are a Perfect RL Showcase

A drone is a highly nonlinear, underactuated system with complex aerodynamics, prop wash effects, and disturbances. Classical controllers struggle at agile flight. RL thrives here.

🔄 Classical Drone Control Stack

// Traditional cascade controller:
Trajectory Planner
  ↓ desired position + velocity
Position Controller (outer loop, ~50 Hz)
  ↓ desired attitude (roll/pitch/yaw)
Attitude Controller (inner loop, ~500 Hz)
  ↓ desired angular rates
Rate Controller (PID, ~1000 Hz)
  ↓ motor commands [ω₁, ω₂, ω₃, ω₄]

This works well for slow, smooth flight. But consider: aggressive maneuvers (powerloops, flips), wind gusts, motor failures, slung loads — the linear assumptions break down.

🚁 RAPTOR — Rapid Agile Post-Stall Trajectory Optimization with RL

📄

Paper Context

Work from Floreano Lab / ETH Zürich group. RAPTOR demonstrated that RL can push quadcopters into aerodynamic regimes completely beyond classical control — including post-stall flight where the rotors stall and classical models completely fail.

Why Classical Control Fails in Agile Flight

  • Aerodynamic nonlinearities: At high velocities/angles, blade element theory breaks down. Prop wash, induced drag — these aren't in the linear model.
  • Post-stall regime: When drone pitches aggressively (>60°), rotors stall like aircraft wings — thrust drops nonlinearly. Classical PID has no model for this.
  • Latency and actuation limits: Motor dynamics are nonlinear at extremes. Classical controller assumes instant response.

The RL Approach in RAPTOR

// State:
s = [pos, vel, quat, ang_vel, // 13-dim
     prev_motor_cmd, progress_indicator]

// Action:
a = [Δω₁, Δω₂, Δω₃, Δω₄] // motor speed residuals

// Reward (tracking + efficiency):
r = -||p - p_ref||² - ||v - v_ref||²
   - 0.01||a||² - penalty_for_crash

Key Technical Contributions

Asymmetric Actor-Critic

Actor (deployed on drone) only uses onboard sensor state. Critic (only during training) sees privileged info like exact drag coefficients. This enables better learning while keeping deployment lean.

Aerodynamic Disturbance Modeling

During training, random aerodynamic forces are injected. The policy learns to implicitly compensate for aero effects it cannot directly model.

Trajectory Conditioned Policy

Policy takes reference trajectory as context — same policy can execute diverse maneuvers (hover, flip, powerloop) by conditioning on different trajectory segments.

Sim-to-Real with DR

Motor time constants, thrust coefficients, drag terms randomized over wide ranges. Policy achieves <10cm tracking error on real hardware doing 14 m/s agile maneuvers.

Phase 1 — Foundations (1-2 months)

Sutton & Barto "Reinforcement Learning: An Introduction" (free online). Complete OpenAI Gym/Gymnasium tasks: CartPole (PPO), MountainCar, LunarLander. Code Q-learning from scratch.

Phase 2 — Deep RL (1-2 months)

Implement DQN, PPO, SAC from scratch in PyTorch. Use Stable-Baselines3 for benchmarking. Read the original papers: DQN (Mnih 2015), PPO (Schulman 2017), SAC (Haarnoja 2018).

Isaac Gym / Isaac Lab (NVIDIA) or MuJoCo with dm_control. Train a legged robot to walk. Implement domain randomization. Do your first sim-to-real (even on a simple arm).

Phase 4 — Drone/Aerial RL (2-3 months)

AirSim or Flightmare simulator. Implement PPO for hover, then trajectory tracking. Study RAPTOR and Swift papers. Attempt deployment on real hardware (Crazyflie is a good starter platform).

Phase 5 — Research (ongoing)

Follow CoRL, ICRA, IROS, NeurIPS (robot learning track). Explore model-based RL (Dreamer), offline RL, world models, foundation models for robotics (RT-2, π₀).
